<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS → app process | OS kill mid-Drift-write can leave a partial transaction — handled by SQLite ACID guarantees; relevant only when Plan 04 writes rounds |
| Local filesystem → Hive | Hive box files are user-writable on the device; untrusted process with device access could tamper |
| Local filesystem → Drift SQLite | Same as Hive; SQLite file `brdy.sqlite` in app documents directory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | Drift SQLite file at `getApplicationDocumentsDirectory()/brdy.sqlite` | accept | Local-only app per CLAUDE.md (no accounts, no PII beyond handicap + course names); device-level threat model is out of scope; SQLite uses `PRAGMA foreign_keys = ON` for referential integrity |
| T-01-02-02 | Information Disclosure | Hive `course_cache` box contains JSON-encoded course data including GPS coordinates | accept | Course GPS data is public-domain (golfcourseapi.com); no user secrets stored; handicap_index is the most sensitive value and is a public WHS number |
| T-01-02-03 | Denial of Service | FMTC store creation on every launch (P-03) | mitigate | `try { ... } catch (e) { Logger().w(...); }` wraps the call so a second-launch failure does not crash the app — app continues without pre-caching |
| T-01-02-04 | Tampering | `HoleOutcome` persisted as free-text string `'double'` (NOT `'doubleBogey'`) | mitigate | Phase 2 will introduce a strict enum-string mapper helper; Plan 01-02 only defines the schema, no rows written yet; outcome column is nullable in Phase 1 |
| T-01-02-05 | Repudiation | `RoundDao.completeRound` records `completedAt` from `DateTime.now()` on the device | accept | Device clock is trusted within the local-only app boundary; no audit log required (no multi-user, no server) |
| T-01-02-06 | Elevation of Privilege | Drift migration `onUpgrade` empty in v1 — future schema bumps could silently skip migrations if executor forgets to add cases | mitigate | `MigrationStrategy` scaffold with the `if (from < 2)` comment in `onUpgrade` is the documented pattern (CLAUDE.md); Phase 2+ executors MUST add cases and re-dump `drift_schemas/` per CLAUDE.md rules |
| T-01-02-SC | Tampering | npm/pip/cargo installs | accept | Plan 02 installs zero new packages — only uses pubspec deps already audited |

**Residual risk:** `drift_schemas/1.json` is committed but never verified against migrations in Plan 01-02 (no v2 yet). The first migration test will land in Phase 2 — Phase 2 executor MUST `dart run drift_dev schema generate drift_schemas/ test/generated_migrations/` and add migration tests at that point. Phase 1 only locks the v1 snapshot.
