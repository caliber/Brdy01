  - `setFairwayHit(bool? fairwayHit)`: same pattern as setPutts.
  - `setGir(bool? gir)`: same pattern, writes greenInRegulation column.
  - `undoOutcome()`: reads current, writes outcome: const Value(null) and recordedAt: const Value(null), then invalidateSelf. Does NOT move activeHoleIndexProvider — that is the screen's responsibility.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Widget → HoleScoreNotifier | User tap enters the notifier; malformed input (negative putts, invalid outcome string) must not corrupt Drift |
| Drift ← HolesCompanion | All writes go through insertOrUpdateHole; concurrent writes on same (roundId, holeNumber) could race |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | HoleDao.insertOrUpdateHole | mitigate | SELECT-then-branch is non-atomic but acceptable: single-user app, sequential taps; no concurrent writers |
| T-02-02 | Denial of Service | runningScoreProvider — byName parse | mitigate | HoleOutcome.values.byName throws ArgumentError on unknown string; wrap in try/catch or guard — if an unknown outcome string is ever stored, provider will error; acceptable since only HoleScoreNotifier writes outcome strings |
| T-02-03 | Information Disclosure | courseJson in Drift rounds row | accept | Local device storage; no PII beyond course name; no network exposure |
| T-02-04 | Elevation of Privilege | ref.read vs ref.watch in notifier | mitigate | Use ref.read(appDatabaseProvider) in all async mutation methods — never ref.watch; following pattern from round_setup_notifier.dart |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | Phase 2 installs zero new packages — no package legitimacy gate required |
</threat_model>
