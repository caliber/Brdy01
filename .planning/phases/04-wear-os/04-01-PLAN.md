Create `WearDataBridgeService.kt` in package `com.brdy.brdy01`. This Android Service handles `/brdy/score` DataItems when the Flutter app is **backgrounded** (phone in bag). Without this, WEAR-02 only works when the phone app is foregrounded.

```kotlin
package com.brdy.brdy01

import android.content.Intent
import com.google.android.gms.wearable.DataEvent
import com.google.android.gms.wearable.DataEventBuffer
import com.google.android.gms.wearable.DataMapItem
import com.google.android.gms.wearable.WearableListenerService
import io.flutter.embedding.engine.FlutterEngineCache

class WearDataBridgeService : WearableListenerService() {
    override fun onDataChanged(events: DataEventBuffer) {
        for (event in events) {
            if (event.type == DataEvent.TYPE_CHANGED &&
                event.dataItem.uri.path?.startsWith("/brdy/score/") == true) {
                val map = DataMapItem.fromDataItem(event.dataItem).dataMap
                val holeIndex = map.getInt("holeIndex")
                val outcome = map.getString("outcome") ?: continue
                val timestamp = map.getLong("timestamp")
                // Forward to active Flutter engine if available (phone app foregrounded)
                val engine = FlutterEngineCache.getInstance().get("main")
                if (engine != null) {
                    val args = mapOf("holeIndex" to holeIndex, "outcome" to outcome, "timestamp" to timestamp)
                    engine.dartExecutor.binaryMessenger.let { messenger ->
                        // Re-use the existing EventChannel sink via LocalBroadcastManager
                    }
                }
                // Always broadcast locally so any active WearDataBridgePlugin picks it up
                val intent = Intent("com.brdy.brdy01.WEAR_SCORE")
                intent.putExtra("holeIndex", holeIndex)
                intent.putExtra("outcome", outcome)
                intent.putExtra("timestamp", timestamp)
                androidx.localbroadcastmanager.content.LocalBroadcastManager
                    .getInstance(this)
                    .sendBroadcast(intent)
            }
        }
    }
}
```

Register in `android/app/src/main/AndroidManifest.xml` inside the `<application>` block:

```xml
        <data android:scheme="wear" android:host="*" android:pathPrefix="/brdy/score/" />
    </intent-filter>
</service>
```

In `WearDataBridgePlugin.kt`, register a `LocalBroadcastManager` receiver in `onAttachedToEngine` that listens for `"com.brdy.brdy01.WEAR_SCORE"` broadcasts and forwards them to the Flutter EventChannel sink. Unregister in `onDetachedFromEngine`. This ensures scores from a backgrounded `WearDataBridgeService` reach the active Flutter engine once the app is foregrounded.

Add `implementation 'androidx.localbroadcastmanager:localbroadcastmanager:1.1.0'` to `android/app/build.gradle` dependencies if not already present.

Do NOT add `FlutterEngineCache` prewarming in this task — the LocalBroadcastManager bridge is sufficient: `WearDataBridgeService` fires the broadcast, `WearDataBridgePlugin` relays it to Flutter when the engine is active. The sequence is: score arrives → Service broadcasts → Plugin EventChannel sink fires → Dart `scoreEvents` stream emits → `_handleWearScore` writes to Drift.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| phone Flutter → phone Kotlin plugin | MethodChannel arguments are trusted (same process), but malformed args must not crash the JVM. |
| phone Kotlin → Google Play Services | DataItems written here are visible to any app on the watch with same applicationId + signing key. |
| keystore file on disk | Single shared secret; loss = inability to update phone or watch app together. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | DataItems on the wire | mitigate | Same-applicationId + same-signing-key gate (enforced by Wearable Data Layer). Keystore handling documented in Task 1. |
| T-04-02 | Denial of Service | rapid putDataItem calls | mitigate | Path includes timestamp `/brdy/hole` already deduped per-write via `ts` field; future score path uses `/brdy/score/{hole}-{ts}` to avoid overwrites. |
| T-04-03 | Information Disclosure | keystore file checked into git | mitigate | `.gitignore` updated to exclude `*.jks` and `key.properties`. |
| T-04-04 | Repudiation | failed DataItem put silently lost | mitigate | `pushHoleState` returns `false` and logs on PUT_FAILED; UI may surface in later plan. |
| T-04-SC | Tampering | gradle dependency `play-services-wearable:18.1.0` | mitigate | Official Google library, verified via developer.android.com/training/wearables/data-layer. No SLOP packages introduced. |
</threat_model>
